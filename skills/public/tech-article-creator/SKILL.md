# 技术文章全流程创作

从选题到发布的生产管线。每个阶段有明确输入、产出和质量门禁。支持从任意阶段切入。

## 配置

文章根目录通过参数指定或从当前工作目录推断。首次使用时确认路径。

```
{根目录}/
├── 选题/                           # 选题文件
├── 微信公众号发布注意事项.md          # 公众号合规规范（如有）
├── 序号 {N}-{标题}/                 # 单篇文章目录
│   ├── 大纲.md                     # 含 [Blog]/[WeChat] 双版本标注
│   ├── 原始素材/                    # 素材 + 评审意见
│   ├── cover-image/                # 配图
│   ├── 【知识星球版】{标题}.md       # Blog 技术深度版
│   ├── 【公众号版】{标题}.md         # 公众号故事版
│   └── 【公众号版】{标题}.html       # 微信排版（自动生成）
```

## 阶段总览

| # | 阶段 | 产出 | 门禁 |
|---|------|------|------|
| 1 | 选题 | 选题推荐（含优先级） | 用户确认 |
| 2 | 大纲 | 大纲.md + 文章目录 | 双版本标注完整 |
| 3 | 素材收集 | 原始素材/*.md | 覆盖大纲所有章节 |
| 4 | 大纲评审 | 评审意见 → 更新大纲 | 采纳后大纲定稿 |
| 5 | 写作+内部评审 | Blog版 + 公众号版（评审通过） | Agent Team 迭代通过 |
| 5.5 | 标题定稿 | 两版最终标题 | 用户选定 |
| 6 | 外部终稿评审 | Codex/Gemini 评审通过 | 多轮迭代通过 |
| 7 | 人工终审 | 用户确认终稿 | 用户 approve |
| 8 | 配图 | 封面图 + 示意图 + 截图 | 中文标注、图片齐全 |
| 9 | 排版/发布 | Blog 发布 + 公众号草稿 | 双平台发布完成 |

## 阶段 1：选题

1. **扫描已有文章**：读取根目录下 `序号 *` 文件夹，提取系列脉络
2. **采集热点源**：本地 digests 目录（近 7 天） > WebSearch > ai-news-digest skill
3. **交叉分析**：已有文章 × 热点 → 系列缺口 / 热点卡位 / 长尾价值
4. **输出 3-5 个选题**：每个含标题、热点依据、衔接关系、核心内容
5. **保存**到 `选题/week{N}-{日期}.md`，用户确认后确定序号

---

## 阶段 2：大纲

1. 创建 `序号 {N}-{标题}/` 目录（含 `原始素材/`）
2. 撰写大纲.md：元信息 + 各章节结构 + **每节标注 `[Blog]`/`[WeChat]`** + 素材清单 + 配图清单

**双版本核心差异**——大纲阶段就要规划：

| Blog 版 | 公众号版 |
|---------|---------|
| 完整配置代码 | 故事 + 类比 |
| 详细成本表 | "多花几美分，少踩一次雷" |
| 技术案例全流程 | 三段式故事（原结论→发现→危险） |
| 复制即用 | 3 条可操作建议 + 引流 blog |

详细写作规范见 `references/dual-version-writing-guide.md`。

---

## 阶段 3：素材收集

启动**并行 subagent** 按主题分工收集，典型分组：
- 个人实战素材（真实案例、配置、工作流）
- 工具/技术素材（功能、对比、配置方式）
- 行业趋势素材（热点新闻、定价、论文）

每个 subagent 产出 `原始素材/{分类}素材.md`，含原始内容和来源标注。收集完检查：每个大纲章节是否都有素材支撑。

详细指南见 `references/material-collection-guide.md`。

---

## 阶段 4：大纲评审

写作前先对大纲+素材做外部评审，确保方向正确再动笔。

将**大纲 + 全部素材文件路径**发送给外部模型。不发摘要——让评审模型读完整文件。

| 模型 | 角色 | 重点 |
|------|------|------|
| Codex (GPT) | 对抗审查员 | 事实错误、逻辑漏洞、定价一致性、脱敏彻底性 |
| Gemini | 编辑顾问 | 双版本策略、标题吸引力、读者行动路径 |

结果处理：保存到 `原始素材/评审意见-{模型}.md`，汇总关键发现（表格+严重度），用户确认后更新大纲。限流时由用户决定是否降级或跳过。

详细模板见 `references/review-prompt-templates.md`。

---

## 阶段 5：写作 + 内部评审（Agent Team）

这是核心阶段，使用 **Agent Team** 实现并行写作和多轮迭代评审。

### Team 编制

| Teammate | 角色 | 职责 |
|----------|------|------|
| blog-writer | Blog 写手 | 撰写技术深度版 |
| wechat-writer | 公众号写手 | 撰写故事认知版 |
| tech-reviewer | 技术评审员 | 审两篇：事实、定价、代码、脱敏、逻辑 |
| content-reviewer | 内容评审员 | 审两篇：AI痕迹、可读性、差异化、行动路径 |

### 执行流程

1. **创建 Team**（4 个 teammate），设置任务依赖：评审任务 blocked by 写作任务
2. **Writer 并行写作**：两个 writer 同时启动，各自读取大纲（注意 `[Blog]`/`[WeChat]` 标注）和素材
3. **补充规范**：公众号 writer 和两个 reviewer 需额外接收微信公众号发布注意事项（如根目录下有 `微信公众号发布注意事项.md`，发给相关 teammate）
4. **初稿完成** → 两个 reviewer 开始评审
5. **迭代循环**：
   - reviewer 读取文章，输出评审意见（问题列表+严重度），发给对应 writer
   - writer 修改后通知 reviewer 再审
   - **迭代直到两个 reviewer 都认为通过**（无高/中严重度问题）
6. **Team 解散**：所有任务 completed 后，shutdown 所有 teammate，删除 team

### Writer Prompt 要点

- 必须读完大纲和所有素材再动笔
- Blog 版：完整配置代码、成本表、结构化案例，4000-5000 字
- 公众号版：翻车故事开头、类比驱动、短段落、引流 blog，2500-3000 字
- 不要 AI 味（"值得注意的是"、过度排比、空洞连接词）
- 案例脱敏："某项目"、通用化表达

### Reviewer Prompt 要点

tech-reviewer 评审维度：
- 事实错误、定价一致性（Opus $5/$25 等）
- 代码可执行性（Blog版）、案例脱敏彻底性
- 逻辑漏洞、成本计算数学正确性

content-reviewer 评审维度：
- AI 痕迹检测
- 双版本差异化是否到位（不是简单删减）
- 公众号版非技术读者可读性、段落长度
- 标题吸引力、读者行动路径
- 微信公众号合规（无品牌 Logo、无诱导文案、来源标注）

详细写作模板见 `references/dual-version-writing-guide.md`。

---

## 阶段 5.5：标题定稿

Agent Team 内部评审通过后、外部终稿评审之前，确定两个版本的最终标题。

### 流程

1. 基于两篇终稿内容，为每个版本生成 5-7 个标题备选
2. 用 AskUserQuestion 让用户选择（分别为 Blog 和公众号各选一个）
3. 用户选定后更新文件标题和文件名

### 标题规则

**Blog 版**：技术主题 + 实战描述，陈述句。

**公众号版**（严格遵守）：
- 不许提问句式（"...吗？"）
- 不要"不是...而是..."句式
- 使用中性描述的陈述句，通过内容本身制造悬念
- 推荐公式见 `references/dual-version-writing-guide.md`

### 标题更新

用户选定后同步更新：文章 H1 标题 + 文件名（`【公众号版】{新标题}.md`）。

---

## 阶段 6：外部终稿评审

Agent Team 内部评审通过后，将终稿发送给外部模型做最终评审。

### 流程

1. 将两篇终稿文件路径发给 **Codex** 和 **Gemini**（并行）
2. 评审维度与阶段 4 类似，但针对完整正文而非大纲
3. 收到评审意见后修改文章
4. **多轮迭代**：修改后附"已修改项"清单再发评审，方便 reviewer 聚焦变更点
5. 迭代直到所有模型都输出"终稿审查通过"
6. 通知用户进入人工终审

### 常见问题类型（按历史经验排序）

外部终稿评审最常发现的问题：
1. **定价/数据口径不一致**：两版引用同一数据但表述不同（如"不到30美分" vs "$0.31"）
2. **配置步骤错误**：写成手动配置但实际是内置插件，读者照做跑不起来
3. **模型/产品命名不精确**：如"Gemini Flash"应为"Gemini 2.5 Flash"
4. **来源标注缺失**：引用了具体数据但没给出处
5. **成本换算错误**：美元到人民币换算不准确
6. **截图占位符未处理**：`（截图：...）`应改为 HTML 注释或替换为实际图片

### 限流处理

与阶段 4 相同：限流时由用户决定，不自动降级。

---

## 阶段 7：人工终审

外部评审通过后，通知用户进行最终人工审核。

人工终审常见的修改方向（AI 评审不易覆盖的）：
- **标题调整**：用户对标题有个人偏好，AI 建议仅供参考
- **干货密度**：某个版本是否太空洞，需要补充实操内容
- **工具推荐平衡**：是否某个工具介绍过多/过少
- **预告/引流**：是否需要下一篇预告（有时用户自己也不知道下篇写什么）

用户 approve 后进入配图/排版/发布。

## 阶段 8：配图

### 配图分类

| 类型 | 来源 | 说明 |
|------|------|------|
| 架构图/流程图 | `baoyu-image-gen` 生成 | 文章核心示意图，**必须使用中文标注** |
| 封面图 | `baoyu-image-gen` 生成 | 带精简中文标题（2-4字），科技风格 |
| 工具截图 | 开源项目 GitHub 仓库 | 如 CCB 分屏截图从 `assets/` 目录获取 |
| 终端输出截图 | 真实截图 | 需用户提供或实际操作截取，不可 AI 生成 |

### 配图规范

- 示意图文字**必须中文**，模型名（Claude、GPT 等）保留英文
- 封面图带精简标题（如"多模型编排"），不要长标题
- 图片统一存放到 `cover-image/` 目录
- Blog 版在 ASCII 图/流程描述旁插入生成的示意图
- 公众号版选择性插入（工具截图优先，技术示意图按需）

### 图片生成注意事项

- Google Gemini Pro 图片模型文字渲染质量最好，优先使用
- 如 Google API 限流，可用 ZenMux 代理（见 memory）或 DashScope 备选
- DashScope 中文渲染尚可但英文拼写易出错，仅作备选
- 并行生成多张图片提高效率

## 阶段 9：排版 / 发布

### 发布清单

| 平台 | 调用 Skill | 产出 | 注意事项 |
|------|-----------|------|---------|
| Blog | `blog-publisher` | 发布到 blog.deepai.wiki | 含 frontmatter、内链、图片上传 |
| 公众号 | `baoyu-markdown-to-html` → `publish-wechat-article` | 草稿箱 | 先转 HTML 再 API 发布 |
| X/Twitter | `baoyu-post-to-x` / `x-tracker` | 推文 | 可选 |

### Blog 发布要点

- 图片路径格式：`![alt](/blog/{slug}/filename.png)`
- 检查 pubDatetime 不超过服务器当前时间
- 添加「相关阅读」内链（2-3 篇），同时反向更新旧文章
- 更新 `references/internal-links.md` 内链关系表
- 图片文件名不能有空格

### 公众号发布要点

- 先用 `baoyu-markdown-to-html` 转换（默认主题从 EXTEND.md 读取）
- 通过 `publish-wechat-article` skill API 发布到草稿箱
- 封面图必填，摘要不超过 120 字
- 内容图片路径需修复为相对路径（skill 自动处理）
- 发布后到草稿箱预览确认再群发

## 安全与合规

### 敏感数据三原则

跨模型评审时：最小披露、先脱敏再送出、可审计可回滚。

文章涉及真实项目时：项目名脱敏（"某项目"）、技术细节通用化、用户确认脱敏程度后才发布。

### 微信公众号合规

如根目录下有 `微信公众号发布注意事项.md`，写作和评审时必须遵守。核心红线：
- 不用第三方品牌 Logo（文字提及品牌名可以）
- 不出现诱导性营销文案（"送你"、"免费领"、"限时"）
- 不放二维码
- 不要标题党（夸大事实、歪曲内容）
- 引用他人内容标注来源
- 发布前对照自查清单检查
