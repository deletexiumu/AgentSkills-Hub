# 双版本写作指南

本指南适用于"技术深度版 + 通俗故事版"双版本模式。如果用户配置了不同的版本策略，按各版本的 `style` 和 `target_audience` 调整写作规范。

## 技术深度版写作规范

### 结构模板

```markdown
# {标题}

**预计阅读时间：{N} 分钟**

---

## 真实场景引入
（用一个真实案例/翻车故事开头，200-300 字）

## 第一部分：为什么需要这个
（痛点 + 数据支撑 + 行业趋势引用）

## 第二部分：实战工作流（核心）
### 架构/流程图
### 核心案例（完整步骤 + 截图 + 代码）
### 辅助案例（简短展示）

## 第三部分：怎么配置
### 最小可用方案（先跑起来）
### 扩展配置（按需添加）
### 方案对比表

## 第四部分：成本/效果分析
（详细估算表 + 对比）

## 第五部分：适用场景
（该用 / 别折腾 的清单）

## 结尾
（行业趋势呼应 + 系列回顾）
# 注意：不要自作主张加"下一篇预告"，除非用户明确要求
```

### 写作要点

- **先展示结果再讲过程**：读者先看到效果才有动力读配置
- **代码块必须可复制执行**：不要给伪代码
- **定价用官方最新数据**：标注来源和查询日期
- **截图位置预留占位符**：`（截图：{描述}）`

## 通俗故事版写作规范

### 结构模板

```markdown
# {吸引力标题}

**预计阅读时间：{N} 分钟**

---

## 翻车/冲突故事（开头钩子）
（一个非技术人也能理解的故事，300 字以内）

## 为什么会这样？（用类比解释原理）
（全栈工程师 vs 专业团队、自己改作文 vs 找人审稿……）

## 我是怎么解决的（核心故事）
### 原结论 → 审查发现 → 为什么危险
（三段式，每段不超过 3 行）

## 这笔账怎么算
（一句话总结成本收益）

## 你明天就能做的 3 件事
（可操作的建议，不涉及具体配置命令）

## 大趋势（结尾升华）
（行业产品引用 + 一句话总结）

> 想看完整配置？→ [技术版详细链接]
```

### 写作要点

- **标题必须有冲突或悬念**，但不能是提问句式、不能用"不是...而是..."、尽量用中性陈述句
- **不用技术术语**：用通俗替代词解释专有名词
- **类比驱动**：每个技术概念都有生活化类比
- **短段落**：每段不超过 3 行，适配手机阅读
- **引流技术版**：通俗版不放代码，用"想看完整配置→技术版链接"引流
- **干货密度要够**：不能全是故事和类比，要有读者能直接用的实操内容
- **不要加"下一篇预告"**：除非用户明确要求

## 案例改写对照

### 技术案例原文（适合技术版）

> 审查工具通过代码审计发现某脚本将真实文档导入数据表（来源标记为 'real_docx'），且数据浏览器暴露了该 schema。原方案只打算脱敏主表，审查推翻了这个结论。

### 通俗版改写

> **原结论**：AI 说"只需要给这 3 个地方加密"。
>
> **审查发现**：第二个 AI 翻了一遍系统代码，发现还有一条"暗道"——数据会通过另一个入口悄悄流入一张不在加密清单里的表。而且这张表，用户能直接打开查看。
>
> **为什么危险**：正门上了锁，侧门大敞。真实数据照样泄露。

## 标题公式

### 技术版（强调技术 + 实战）
- `{技术主题}实战：{具体做法}`
- `{技术主题}：{从A到B的进化}`

### 通俗版（强调冲突 + 认知）

**标题规则**：
- 不许出现提问句式（"...吗？"、"...？"开头的反问）
- 不要"不是...而是..."句式
- 尽量使用中性描述的陈述句，通过内容本身制造悬念

**推荐公式**：
- `{看似完美的事} + 转折 + {意外发现}`
- `{小代价} + {大收益}`
- `{主语} + {动作} + {意外结果}`
- `{具象场景} + {隐藏风险}`

**避免**：
- ~~提问句式~~
- ~~"不是...而是..."句式~~
